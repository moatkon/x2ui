# X2Post 二级节点信息架构与交互规范

> 范围：公开 Feed、节点目录与详情、发布、搜索、关注。
> 目标：在保留 5 个稳定一级节点高可发现性的同时，支持更精准的内容归类。
> 约束：只支持两级；二级节点不是标签、权限组或独立社区。

## 1. 关键结论

1. 一级节点继续承担社区心智和主导航，二级节点只承担该一级节点内的讨论细分。
2. 层级最多两级，不允许二级节点再有子级；更具体、可跨节点的概念继续使用标签。
3. 每个一级节点建议维护 3–6 个二级节点，硬上限 7 个；少于 3 个时不启用二级筛选。
4. Feed 始终直接显示全部 5 个一级节点；选中一级后，在下一行展开它的二级节点。
5. “全部产品设计”是父级聚合视图，不是虚构的二级节点；父节点自己的帖子也包含在聚合结果中。
6. 二级筛选状态写入 URL，可分享、刷新及通过前进/后退恢复。
7. 发帖必须选一级节点；二级节点为推荐选择但不是强制，避免不合适分类和旧内容迁移问题。
8. 关注父节点表示订阅其全部现有及未来二级节点；用户可单独静音某个二级节点。
9. 只关注二级节点不会自动关注父节点，但父节点入口显示“已关注 1 个二级节点”。
10. 移动端不依赖下拉、悬停或横向滚动；一级和当前二级入口都自然换行并保持 44px 点击目标。

## 2. 分类模型与治理边界

### 2.1 数据关系

- `PrimaryNode`：稳定、数量少、可独立发布内容的一级讨论空间。
- `SecondaryNode`：必须且只能属于一个一级节点，可独立关注和筛选。
- `Post.primaryNodeId`：必填。
- `Post.secondaryNodeId`：可空；非空时必须属于所选一级节点。
- 节点停用后保留历史 URL 和帖子归属，页面显示“已归档”，不允许新发帖。

### 2.2 数量规则

- 全站一级节点：当前固定 5 个；新增需要信息架构评审。
- 单个一级节点二级节点：推荐 3–6 个，硬上限 7 个。
- 二级节点少于 3 个时，优先用标签，不显示单独的二级筛选区。
- 某二级节点连续 90 天内容极少时先合并或归档，不继续细分。
- 某二级节点长期承载多个互不相关的用户任务时，可拆分，但拆分后仍不得出现第三级。
- 二级节点名称在同一父节点内唯一；长度建议 2–8 个汉字，避免重复父节点名称。

### 2.3 节点与标签的区别

| 维度 | 二级节点 | 标签 |
| --- | --- | --- |
| 归属 | 只能属于一个一级节点 | 可跨节点使用 |
| 数量 | 少且受治理，3–7 个/父节点 | 多，由内容逐步形成 |
| 用途 | 决定内容进入哪个稳定讨论流 | 描述技术、场景、方法等具体主题 |
| 关注 | 可关注、静音 | 可搜索，不继承节点关注 |

## 3. 五个一级节点的建议二级节点

| 一级节点 | 二级节点示例 | 范围提示 |
| --- | --- | --- |
| 产品设计 `product` | 产品策略 `strategy`、用户研究 `research`、交互与原型 `interaction`、设计系统 `design-system`、增长与实验 `growth` | 产品判断、研究方法及体验方案 |
| 前端开发 `frontend` | HTML/CSS `html-css`、JavaScript `javascript`、框架与应用 `frameworks`、工程与测试 `engineering`、性能与无障碍 `quality` | 浏览器端实现、质量和工程实践 |
| 社区运营 `operations` | 冷启动 `launch`、内容运营 `content`、成员成长 `member-growth`、社区健康 `community-health`、治理与安全 `governance` | 社区生命周期、秩序和成员体验 |
| 写作交流 `writing` | 非虚构写作 `nonfiction`、创意写作 `creative`、编辑与反馈 `editing`、知识整理 `knowledge`、写作习惯 `practice` | 创作过程、表达训练和作品反馈 |
| 生活方式 `life` | 居家与整理 `home`、效率与节奏 `rhythm`、饮食与手作 `food-craft`、城市与出行 `city-travel`、兴趣与休闲 `hobbies` | 低风险的日常经验和兴趣交流 |

这些是首轮验证用分类，不应以热度自动生成。节点管理者根据错投率、搜索词和内容重叠度做季度评审。

## 4. Feed 行为

### 4.1 默认状态

- 一级导航仍显示：“全部、产品设计、前端开发、社区运营、写作交流、生活方式”。
- “全部”显示全站公开 Feed，不显示二级导航，避免用户误以为二级节点可跨父级组合。
- 帖子元信息显示 `一级节点 · 二级节点`；没有二级归属时只显示一级节点。

### 4.2 选择一级节点

- 点击“产品设计”后更新父级结果，并在其下展开二级区：
  `全部产品设计、产品策略、用户研究、交互与原型、设计系统、增长与实验`。
- “全部产品设计”包含该父节点直属帖子及其所有二级节点帖子。
- 当前一级与二级都通过文字、背景和 `aria-current="page"` 表达，不只依赖颜色。
- 每个可点击项自身直接承担背景、内边距和焦点环，不用外层空白链接包裹内层选中块。
- 切换一级节点时清除旧的二级参数，并展开新一级节点的二级区。
- 再次点击当前一级节点不收起二级区，避免界面结构跳动。

### 4.3 选择二级节点

- 点击“用户研究”只显示 `product/research` 内容。
- 结果标题显示“产品设计 / 用户研究”，并提供“一键返回全部产品设计”。
- 页码重置为第一页；浏览器返回应恢复节点、页码和合理的滚动位置。
- 二级节点没有公开帖子时保留筛选和节点说明，不自动跳回父级。

## 5. URL 规范

```text
# Feed 聚合
#/feed
#/feed?node=product
#/feed?node=product&subnode=research

# 节点目录与详情
#/nodes
#/nodes/product
#/nodes/product/research

# 搜索结果
#/search?q=%E7%94%A8%E6%88%B7&type=posts&node=product&subnode=research

# 发布入口可预填，但提交时仍需确认
#/compose?node=product&subnode=research
```

- `node` 与 `subnode` 使用稳定 slug，不使用中文名称或数据库自增 ID。
- `subnode` 无法归属于 `node` 时忽略二级参数并显示非阻塞提示。
- 旧的 `#/feed?node=product` 和 `#/nodes/product` 保持有效。
- 节点重命名不改变 slug；合并后旧 URL 以页面内说明引导到新节点。

## 6. 节点目录与详情

### 6.1 节点目录 `#/nodes`

- 一级节点按稳定运营顺序排列，不按热度或个性化重排。
- 每个一级节点块显示简介、帖子数、关注状态，以及全部二级节点链接。
- 二级节点显示一行范围说明；首屏不展示排行榜和“热门”排序。
- 一级块底部提供“查看全部产品设计帖子”和“了解节点规则”。

### 6.2 一级详情 `#/nodes/product`

- 顶部显示一级节点名称、简介、规则和“关注全部产品设计”。
- 主体直接展示其二级节点列表，而不是把二级入口藏进下拉框。
- 内容区默认显示父级聚合 Feed，并可用同一套二级导航筛选。
- 若用户只关注部分二级节点，关注状态写为“已关注 2 个二级节点”。

### 6.3 二级详情 `#/nodes/product/research`

- 面包屑为“节点 / 产品设计 / 用户研究”。
- 显示二级节点说明、适用/不适用内容示例、父节点共同规则和额外规则。
- 提供“关注用户研究”“浏览帖子”“发布到此处”三个明确动作。
- 不重复显示与父级完全相同的关注人数，统计口径需明确为二级节点自身。

## 7. 发布与搜索

### 7.1 发布

- 表单先选择一级节点，再显示该父级的二级节点单选列表或原生 Select。
- 一级节点必填；二级节点标注“建议选择，无法判断时可留空”。
- 二级选项显示名称和一句范围提示，不能只有难以理解的短名称。
- 修改一级节点后清空已选二级节点，并用礼貌提示说明需要重新选择。
- 从二级详情点击“发布到此处”时预填两级，但确认发布页仍完整展示归属。
- 发布成功后的帖子元信息和返回链接保留两级归属。

### 7.2 搜索

- 搜索筛选先选一级，再显示其二级；不提供跨父级多选二级节点的复杂组合。
- 搜索结果中的节点匹配同时覆盖一级/二级名称与简介。
- 节点结果分组展示父子关系，例如“用户研究 · 产品设计”。
- 清除一级筛选时同时清除二级筛选；清除二级时保留一级。
- 零结果页保留查询词与节点条件，并提供“仅移除二级筛选”和“清除全部筛选”。

## 8. 关注与继承

- 关注一级节点：Feed 订阅其直属帖子、全部当前二级节点及未来新增二级节点。
- 新增二级节点进入父级关注 Feed，但默认不增加高频通知；设置页说明这一差异。
- 静音某二级节点：覆盖父级继承，该二级内容不进入关注 Feed；父级其余内容不受影响。
- 单独关注二级节点：只订阅该二级，不自动关注父级或兄弟节点。
- 同时关注父级和二级时只生成一份内容，不重复展示或通知。
- 取消关注父级时保留用户主动关注的二级节点，并在确认文案中列出保留项。
- 父级关注按钮必须写明“关注全部产品设计”；二级按钮写明“关注用户研究”。

## 9. 320px 与响应式要求

- 一级和二级导航均使用自然换行，整页不得产生横向滚动。
- 每个入口至少 44×44px；选中背景均匀包裹文字，左右内边距相等且不被父元素额外空白放大。
- 一级区和二级区有可见标题与间距，如“按一级节点浏览”“产品设计下的主题”。
- 二级名称不截断；过长时允许单项内部换行，但不缩小到 12px 以下。
- 二级区只展示当前父级的 3–7 项，不同时铺开所有父级的二级节点。
- 200% 缩放下焦点项可见，固定导航不能遮住内容；目录卡片降为单列。

## 10. 无障碍

- 一级、二级区分别使用带可见标题的 `nav`；链接使用真实 `href`，支持新窗口和历史记录。
- 当前两级链接分别使用 `aria-current="page"`；不可用节点使用说明文字，不用空链接。
- 键盘顺序为一级导航、当前二级导航、结果摘要、帖子列表。
- 焦点样式与选中样式同时存在且可区分；不得只靠黑色填充表达状态。
- 结果更新使用 `aria-live="polite"` 播报“已显示用户研究，共 12 篇帖子”，不强制移动焦点。
- 层级关系通过面包屑和可读文本表达，不依赖缩进、字号或颜色。
- 触发异步加载时保留控件尺寸，并向辅助技术暴露忙碌状态。

## 11. 空状态与异常状态

| 场景 | 文案与动作 |
| --- | --- |
| 二级节点无帖子 | “用户研究暂时没有公开帖子。”；查看节点说明、返回全部产品设计、发布新主题 |
| 搜索无结果 | 保留关键词；移除二级筛选、清除全部条件 |
| 二级节点归档 | 说明归档日期和去向；浏览历史内容、前往父节点或合并后的节点 |
| URL 父子不匹配 | “该主题不属于所选节点，已显示产品设计全部内容”；清除无效参数 |
| 加载失败 | 保留当前两级选中状态；重试，不跳回全部 Feed |
| 父节点没有二级节点 | 不显示空导航；说明该节点暂未细分 |
| 发布时二级节点停用 | 阻止提交，保留草稿并要求重新选择二级或留空 |

## 12. 验收测试

- [ ] Feed 首屏无需点击即可看见全部 5 个一级节点。
- [ ] 点击任一一级节点后，一次交互内可见它的全部 3–7 个二级节点。
- [ ] 当前一级和二级的选中背景都紧贴自身点击区域，四周内边距均匀。
- [ ] `node` 与 `subnode` 可刷新、分享、前进和后退恢复；旧一级 URL 不失效。
- [ ] 从产品设计切到前端开发会清除 `subnode=research`，不会显示空错误结果。
- [ ] “全部产品设计”包含直属帖子和所有二级帖子，二级筛选只包含自身帖子。
- [ ] 帖子列表、详情、搜索结果和发布确认均一致显示两级归属。
- [ ] 关注父级后可收到所有子级内容；静音一个子级后其内容被排除。
- [ ] 只关注一个子级不会把父级按钮错误显示为“已关注全部”。
- [ ] 父级与子级同时关注时，关注 Feed 和通知没有重复项。
- [ ] 发布切换父级会清空不再有效的二级选择，草稿正文不丢失。
- [ ] 搜索清除二级条件后保留一级条件，清除一级时同步清除二级。
- [ ] 二级为空、归档、加载失败和父子参数不匹配都有可恢复动作。
- [ ] 320px 和 200% 缩放下无横向滚动，节点名称不截断，点击目标不小于 44px。
- [ ] 仅用键盘可按一级→二级→结果顺序操作，焦点清晰且状态由读屏正确读出。
- [ ] 节点目录、一级详情和二级详情均可互相返回，不存在只能从搜索抵达的孤立页面。
