# X2Post 轻发布严格减法 UX 规范

> 角色：UX Researcher  
> 日期：2026-07-13  
> 依据：`docs/ux-low-cost-creation.md`、当前 `prototype/assets/app.js` 的 `quickComposePage()`  
> 目标：让用户从 Feed 到发布一条有上下文的内容，除输入外只需要 3 次激活；保留不可变发布的知情确认，但移除流程感、配置感和 AI 工具感  
> 范围：极简 P0；不修改原型代码

## 1. 审查结论

当前轻发布虽然每个组件都有合理解释，但组合后已经接近一张完整发布表单：

- Feed 同时出现页头“轻发布”、四张入口卡、重复的“开始轻发布”和“写长帖”。
- 轻发布页先让用户看 Breadcrumb、页头、说明、三步 Steps，再看五张选择卡。
- 内容区同时展示提示、帮助、正文、计数、资料链接、三个 AI 动作、保存草稿和 AI 结果区。
- 节点区再次要求处理两个 Select、路径、可选背景和不可变提示。
- 最终确认又加入金币与共建说明。

这些内容单独看都正确，但用户的真正任务只是：**把刚才脑中的一句话公开留下。** 当前页面却要求用户先理解“产品如何支持轻发布”。

极简 P0 应改成“一页一件事”：

```text
确认我正在说什么、发到哪里
→ 写一句
→ 最后看一眼
→ 发布
```

表达形态和节点优先从 Feed 上下文继承，不再把它们当作进入页面后必须重新完成的步骤。AI、资料链接和节点修改都是例外路径，只有用户主动需要时才展开。

## 2. 极简线框

### 2.1 Feed 入口

现有大 Panel、四张说明卡、页头重复按钮应缩成一条轻量输入入口：

```text
┌──────────────────────────────────────────┐
│ 说点具体的……                             │
│ [问问题] [分享做法] [贴资料] [请大家判断] │
└──────────────────────────────────────────┘
```

- “说点具体的……”进入默认“问问题”。
- 四个动作只显示短标签，不在 Feed 展示帮助说明、预计用时、字数或不可变规则。
- 当前一级/二级节点自动进入 URL；用户不需要再次选择。
- 页头只保留一个发布入口。若此条入口存在，删除页头里重复的“轻发布”按钮。
- “写长帖”降为轻入口末尾的普通文本链接，不与四个轻动作争夺主视觉。

### 2.2 轻发布页

```text
← 返回 Feed                         发一个问题  [换一种]

发布到 产品设计 / 用户研究                         [修改]

你想请大家帮什么？
┌──────────────────────────────────────────┐
│ 例如：怎样让第一次发帖的人选对节点？     │
│                                          │
└──────────────────────────────────────────┘
0 / 280

[＋添加资料链接]        [写不顺？AI 可帮你理顺]

发布前会显示完整内容
                                             [发布]
```

资料模式的唯一差异：来源 URL 是必要内容，因此直接出现在正文下方，不再经过“添加资料链接”展开。

```text
这份资料具体能解决什么？
[ textarea ]
来源链接
[ https:// ]
```

### 2.3 最终确认

```text
最后看一眼

┌──────────────────────────────────────────┐
│ [一个问题] [AI 协助整理，仅存在时显示]    │
│ 怎样让第一次发帖的人在一分钟内选对节点？ │
│ 发布到：产品设计 / 用户研究              │
└──────────────────────────────────────────┘

确认后内容不能编辑或删除；需要修正时可另发补充。

                         [继续修改] [确认发布]
```

确认层只回答两件事：“我将发布什么”和“发布后会怎样”。金币、共建、AI 原理、失败保留、节点继承规则都不在这里再次教育。

## 3. 首屏必须保留、删除和按需展开

### 3.1 必须保留

| 元素 | 最小呈现 | 原因 |
|---|---|---|
| 返回 | “返回 Feed” | 提供无损退出路径 |
| 当前形态 | “发一个问题” + “换一种” | 让提示语有语义，但不重做完整选择流程 |
| 发布位置 | `产品设计 / 用户研究` + “修改” | 不可变内容必须避免发错位置 |
| 可见 label | 随形态变化的一个具体问题 | placeholder 不能替代 label |
| 正文 | 单一 textarea | 页面唯一主要任务 |
| 字数 | 安静显示 `0 / 280` | 给边界，不做完成度压力 |
| 主动作 | 一个“发布”按钮 | 首屏只允许一个主 CTA |
| 自动保存状态 | 只在保存发生后显示“已保存” | 保留安全感，不要求操作 |

### 3.2 从首屏删除

| 当前元素 | 决策 | 理由 |
|---|---|---|
| Breadcrumb | 删除 | 当前任务短且来源明确；返回链接已足够 |
| 页头说明和右侧“改写长帖”按钮 | 删除 | 与形态标签、Feed 入口重复；长帖改为次级文本链接 |
| “三步完成” Steps | 删除 | 三步提示把一个短动作包装成流程 |
| 四张格式 Card + 自由发布 Card | 删除 | Feed 已选形态；大卡迫使用户重新决策 |
| 编号 1/2/3 Panels | 删除 | 强化表单长度和任务负担 |
| 格式帮助段落 | 删除 | 保留一个具体 label 和 placeholder 即可 |
| “约 1 分钟” Badge | 删除 | 时间承诺不是完成任务所需信息 |
| “先写原意，不需要润色” | 删除 | 可放进首次 placeholder，不需常驻 |
| 三个并列 AI 按钮 | 删除 | 让用户在写作前先选择工具，增加决策成本 |
| “保存这句草稿”按钮 | 删除 | 自动保存，不把系统责任变成用户任务 |
| 可选背景字段 | 删除 | 用户可直接写进正文；单独字段制造“还缺一项”的感受 |
| 一级/二级节点两个 Select | 默认折叠 | 来源节点已知时不应重新选择 |
| Footer 中重复不可变提示 | 删除 | 只在最终确认出现一次完整后果 |
| 确认层金币/共建说明 | 删除 | 与当前决定无关，且制造惩罚感 |

### 3.3 按需展开

| 元素 | 何时显示 | 展开内容 |
|---|---|---|
| 更换形态 | 点击“换一种” | 四个紧凑 radio 选项；选择后立即收起并更新 label/placeholder |
| 修改节点 | 点击“修改” | 先一级、再二级；保存后收起为路径摘要 |
| 资料链接 | 资料模式默认显示；其他模式点击“添加资料链接” | 一个 URL 输入，不再附长说明 |
| AI | 点击“写不顺？” | 空白时只给“回答 3 个问题”；有文字时只给“帮我理顺” |
| AI 结果 | AI 完成后 | 原文保留、建议文本、采用/不用；只有采用后显示公开标签 |
| 长帖 | 页面末尾普通链接 | “需要标题、附件或 Markdown？改写长帖” |

## 4. 最少动作路径

### 4.1 从已筛选 Feed 发普通轻内容

```text
1. 点击 Feed 的“问问题 / 分享做法 / 请大家判断”
2. 输入正文
3. 点击“发布”
4. 在确认层点击“确认发布”
```

除输入外为 3 次激活。类型和一级/二级节点来自入口 URL，不要求额外选择。

### 4.2 从 Feed 发资料

```text
1. 点击“贴资料”
2. 输入自己的判断
3. 输入来源 URL
4. 点击“发布”
5. 点击“确认发布”
```

资料链接是内容本身，不应为了减少一次输入而隐藏或自动猜测。

### 4.3 从全局导航进入

全局“轻发布”没有可靠节点上下文时，不能静默默认到数组中的第一个节点。优先级：

1. 使用当前 Feed 已选的节点/子节点。
2. 没有时使用用户最近一次成功发布的位置，并明确显示。
3. 两者都没有时，节点修改区初始展开并要求选择一级节点；子节点仍推荐但非必选。

全局入口可以默认“一个问题”，用户可通过“换一种”修改。默认值必须显示，不能隐藏类型判断。

## 5. 不可变确认：保留知情，不制造焦虑

### 5.1 保留什么

- 点击“发布”后必须出现一次最终确认。
- 确认层展示完整正文，不用截断摘要。
- 展示节点路径、资料 URL 和 AI 标签，但只展示实际存在的项目。
- 明确说明“确认后内容不能编辑或删除”。
- 有“继续修改”和“确认发布”两个动作。
- 双击确认只生成一次内容；提交中按钮禁用并显示状态。

### 5.2 删除什么

- 不使用强制 checkbox 或要求用户再输入“确认”。
- 不用三次重复的 warning Alert。
- 不在编辑页、按钮旁和 Modal 中重复同一长句。
- 不写“撤回、金币、共建奖励、AI 责任、失败恢复”五件事组成的警告段落。
- 不使用倒计时、红色危险按钮或阻吓性标题“你确定吗？”。

### 5.3 推荐文案

- 编辑页主动作下：`发布前会显示完整内容。`
- 确认层标题：`最后看一眼`
- 后果：`确认后内容不能编辑或删除；需要修正时可另发补充。`
- 次动作：`继续修改`
- 主动作：`确认发布`

“不能撤回”可以保留在站点规则和长帖完整须知；轻发布确认中“不能编辑或删除”已经表达实际后果。若法律/产品契约要求必须出现“不能撤回”，则使用一句：`确认后内容不能编辑、删除或撤回；需要修正时可另发补充。`，不要再增加第二个警告块。

## 6. AI、节点和资料的渐进披露

### 6.1 AI

AI 不是首屏能力清单，而是写作受阻后的救援：

- 空 textarea：显示入口“写不出来？回答 3 个问题”。
- 已输入内容：显示入口“写不顺？AI 可帮你理顺”。
- P0 删除独立“清晰度检查”。必要的空白、长度、URL、常见联系方式检查作为普通表单校验，不包装成 AI 功能。
- 用户未展开 AI 时，不显示 AI 原理、披露规则或结果容器。
- 采用建议后，只在正文下显示一行“已采用 AI 整理 · 撤回采用”；详细说明放进可展开帮助。
- AI 失败只显示原位提示“没有改动你的草稿”，焦点回到正文。

### 6.2 节点

- 正常态只显示一行路径，不显示 Select。
- 从 Feed 进入时继承当前一级/二级节点。
- 点击“修改”才展开 Select；修改后立即收起。
- 更换一级节点后才刷新二级节点；不要一次展示全部两级列表。
- 最终确认再次显示路径，防止不可变内容误投。
- 父/子节点规则不常驻首屏；仅在用户修改位置或校验冲突时显示相关一句。

### 6.3 资料链接

- 资料形态直接显示 URL，因为它是必填内容。
- 其他形态只有“添加资料链接”安静入口。
- 添加后显示域名和完整可编辑 URL；不自动抓取大预览阻断输入。
- URL 无效时在字段下原位报错；正文保留。
- 来源安全检查在提交时运行；异常时说明原因和恢复动作，不自动删除用户正文。

## 7. Feed 也要减法

若只精简轻发布页，而 Feed 继续显示大 Panel + 四张 Card + Footer CTA，用户仍会在入口处感到“要开始一项任务”。Feed P0 需要同步减法：

- 页头删除“轻发布”和面板 Footer “开始轻发布”中的至少一个；推荐只保留紧凑入口本身。
- 四种形态仅显示短标签，不显示四段 help。
- 删除“约 30–90 秒、最多 280 字、发布前仍会确认”。这些说明不帮助选择。
- 空子节点保留三个上下文 CTA，因为它们替代空内容；但每项只保留动词短语，不重复解释。
- “写长帖”保留为普通链接，确保自由表达没有消失。

## 8. 移动端要求

- 320px 单列；头部“形态 + 换一种”和“节点 + 修改”允许两行，不横向滚动。
- textarea 默认约 5 行，随内容增长到合理上限；不在初始态占满一屏。
- 不使用固定在视口底部、可能遮挡虚拟键盘的发布条。页面足够短，按钮放在表单自然流末尾。
- 虚拟键盘打开时不自动滚动到页面顶部；错误字段使用 `scrollIntoView({block: "center"})` 后聚焦。
- 展开的形态、节点和 AI 区域使用原位披露，不打开多层 Modal。最终确认是唯一必需 Modal。
- 触控目标至少 44×44px；“修改”“换一种”“添加资料链接”即使视觉为文字按钮，也要有足够点击区。
- 资料 URL 使用 `inputmode="url"`；正文使用正常多行输入，支持系统语音输入。
- 返回 Feed 时若已有输入，自动保存并以轻量 Toast 告知；不要再弹一次离开确认，除非自动保存失败。

## 9. 键盘与读屏要求

- 从 Feed 激活具体形态后，轻发布页加载完成将焦点放到 textarea。
- textarea 的可见 label 同时表达形态，例如“你想请大家帮什么？”；一个礼貌状态区播报“发一个问题，发布到产品设计 / 用户研究”。
- 不让逐字符计数 `aria-live` 每次按键都播报；仅在接近上限和超限时播报。
- “换一种”“修改”“添加资料链接”“写不顺？”使用原生 button，并维护 `aria-expanded`、`aria-controls`。
- 展开区出现后，焦点移到第一个新控件；完成或取消后回到触发按钮或正文。
- Radio 形态组使用 `fieldset` 和可见/屏幕可见 `legend`；箭头键遵循原生 radio 行为。
- 最终确认使用原生 dialog 语义；打开后焦点进入标题或“继续修改”，关闭后回到“发布”。
- 确认成功后焦点进入 Feed 的成功状态或新内容标题，不只显示短暂 Toast。
- 错误与字段通过 `aria-describedby` 关联；错误摘要不是必需，因为 P0 同时最多只有正文和 URL 两个字段。
- AI 建议不只用颜色表示变化；建议文本可完整线性阅读。

## 10. P0 状态

极简不代表只设计成功态。P0 至少覆盖：

| 状态 | 表现 |
|---|---|
| 空正文 | 正文下原位提示“先写下一句”，聚焦正文 |
| 超过 280 字 | 计数和文字错误；不截断用户输入 |
| 资料缺 URL | URL 下原位提示，正文保留 |
| URL 无效/风险 | 说明具体类别并允许修改或移除来源 |
| 节点无效/关闭 | 自动展开节点修改，并说明原位置不可发布 |
| AI Loading | 只在展开区显示；正文仍可编辑 |
| AI Error | “没有改动你的草稿” + 重试/关闭 |
| 自动保存成功 | 安静显示“已保存” |
| 自动保存失败 | 明确显示“尚未保存”，提供重试；不阻断继续编辑 |
| 发布中 | 禁用确认按钮、显示“正在发布”，幂等 |
| 发布失败 | Modal 保持打开或返回编辑页；正文、URL、节点和 AI 采用状态完整保留 |
| 发布成功 | 返回来源 Feed，定位新内容并提供可感知成功状态 |

## 11. UI 实现验收标准

### 11.1 页面减法

- [ ] 轻发布首屏不存在 Breadcrumb、Steps、编号 Panels 或大格式 Card 网格。
- [ ] 首屏只有一个主 CTA：“发布”。
- [ ] 普通路径首屏只有一个 textarea；资料路径额外只有一个 URL 输入。
- [ ] 不存在显式保存草稿按钮；保存自动发生。
- [ ] 不存在“补充背景”独立字段。
- [ ] AI 不以三个并列按钮常驻显示。
- [ ] 节点 Select 默认折叠，路径摘要始终可见。

### 11.2 最少动作

- [ ] 从带 type/node/subnode 的 Feed 入口进入后，不要求重新选择类型或节点。
- [ ] 普通轻发布除输入外最多 3 次激活：入口、发布、确认发布。
- [ ] 普通路径不展开 AI、节点或资料也能完成。
- [ ] 全局入口没有上下文时不会静默发布到第一个节点。

### 11.3 不可变确认

- [ ] 最终确认只出现一次，展示完整正文与实际存在的元信息。
- [ ] 明确写出“不能编辑或删除”，并提供修正路径。
- [ ] 不使用 checkbox、倒计时、红色危险 CTA 或重复 warning。
- [ ] 确认层不展示金币、共建奖励或无关规则。
- [ ] 双击/重试确认不会生成重复内容。

### 11.4 渐进披露

- [ ] 更换形态、修改节点、添加资料、使用 AI 均由用户主动展开。
- [ ] 展开后可以无损收起；已输入正文不丢失。
- [ ] 资料形态默认显示 URL；其他形态默认不显示。
- [ ] AI 空白态只提供澄清问题，有文本时只提供整理，不让用户先选三种工具。

### 11.5 移动端与无障碍

- [ ] 320px 无整页横向滚动；200% 缩放内容与 Modal 可用。
- [ ] 虚拟键盘不被固定发布栏遮挡。
- [ ] 所有文字按钮触控区域至少 44×44px。
- [ ] 从 Feed 进入后焦点到 textarea，并播报当前形态和节点。
- [ ] 全流程可用键盘完成；展开控件有 `aria-expanded/controls`。
- [ ] 计数不会逐字符打扰读屏；错误就近且可聚焦。
- [ ] Dialog 关闭后焦点返回触发点；成功后焦点到新内容或成功状态。

## 12. 决策摘要

极简 P0 只保留四件事：

1. **一句话的语义**：问题、做法、资料或判断。
2. **一句话本身**：唯一主要输入。
3. **它被放到哪里**：继承节点，必要时再改。
4. **不可变知情确认**：完整内容最后看一眼。

其他能力全部后退：自动保存不要求点击，AI 在写不顺时出现，资料只在需要时出现，节点只在需要修改时展开，规则只在发生冲突或最终确认时出现。

判断标准不是页面是否显得“功能完整”，而是用户能否在还记得自己那句话的时候把它发出去。
