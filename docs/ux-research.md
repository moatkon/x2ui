# X2Post 轻量级社区 UX 桌面研究与执行建议

> 研究日期：2026-07-13  
> 输入：`轻量级社区产品方案.md`、`轻量级社区_UI设计描述文档.md`  
> 适用范围：当前 P0 + P1；不扩展 P2/P3  
> 目的：为静态 HTML 全页面原型和后续应用实现提供用户旅程、状态、信任安全及验收依据

## 1. 执行摘要

X2Post 的关键体验不是“让用户尽快发帖”，而是让用户用较低风险完成以下迁移：

```text
匿名判断内容是否值得信任
→ 找到与自己相关的节点/帖子
→ 用低成本动作试探参与
→ 在不丢上下文的情况下完成登录
→ 清楚理解不可变规则后贡献内容
→ 收到有意义的回应并回访
→ 遇到伤害时能够屏蔽、举报并获知处理状态
```

对当前 HTML 原型影响最大的 8 条结论：

1. **阅读本身是有效的新手阶段。** 不应在公开 Feed、节点、帖子、搜索或用户页前设置登录墙，也不应用强弹窗打断匿名浏览。
2. **首次参与必须连续。** 匿名用户点收藏、反应、关注、评论或发布时，登录后应回到原页面、原滚动位置，并恢复已输入内容和待执行动作。
3. **不可变发布必须降低误操作风险。** 发帖和评论需要“编辑/预览 → 不可变提示 → 最终确认 → 提交”的闭环；失败保留输入，成功后不出现编辑、删除、撤回入口。
4. **首次回应比更多功能更重要。** 冷启动期应优先确保真实帖子获得及时、有内容的回复，而不是引入推荐、徽章、活动等冻结能力。
5. **关注和通知应服务于回访，不制造压力。** 通知需按回复、提及、互动、关注、审核、安全分型，支持单条/全部已读、偏好设置及失效目标的安全降级。
6. **治理必须可理解、可追踪。** 举报人需要提交确认和状态；被处置者需要知道适用规则、动作、时限和申诉入口；隐藏内容保留上下文占位，不暴露原文。
7. **节点规则应在决策点出现。** 节点页展示完整规则；发帖页和举报/审核场景展示与当前任务相关的规则摘要，避免依赖用户记忆。
8. **原型必须覆盖状态，而不只是成功页面。** 最少覆盖 loading、empty、partial、error、401、403、404、429，以及写操作的 dirty、submitting、conflict、failed。

## 2. 研究方法与证据边界

本研究采用方案审阅、任务分析和公开桌面研究，没有进行 X2Post 真实用户访谈或可用性测试。因此，用户动机与优先级属于待验证假设，不能替代上线后的行为数据和定性访谈。

相关证据给出以下方向：

- 潜水常与学习社区文化、建立关系和安全顾虑有关，不宜简单视为“未贡献”；改善易用性、鼓励和新手指导可帮助参与转化。[Understanding lurkers in online communities: A literature review](https://www.sciencedirect.com/science/article/abs/pii/S0747563214003008)
- 对 2,777 名新闻组新用户的研究发现，首次发帖获得回复者再次发帖的概率从 44% 上升到 56%。[Predicting Continued Participation in Newsgroups](https://academic.oup.com/jcmc/article/11/3/723/4617705)
- 2025 年一项自然实验中，提醒社区成员善待新人的干预使新人内容获得的评论增加 54%，新人再次发布的可能性提高 10%，但未持续改善后续贡献质量。这意味着“欢迎”要与持续指导结合，不能只做一次性标记。[Behavior Toward Newcomers and Contributions to Online Communities](https://aisel.aisnet.org/misq/vol49/iss2/16/)
- Wikimedia 在 27 个非英语社区测试统一的新手首页，只得到有限且依赖情境的提升；X2Post 不应优先建设脱离具体任务的庞大 onboarding 中心。[Wikimedia Research Newsletter: Newcomer Homepage](https://meta.wikimedia.org/wiki/Research%3ANewsletter/2024/March)
- 举报实验表明，在举报界面呈现社区规则会增强透明感，提供自由文本会增强用户表达意见的感受；但仍需控制认知负担。[Incorporating Procedural Fairness in Flag Submissions on Social Media Platforms](https://arxiv.org/abs/2409.08498)
- Santa Clara Principles 将清晰规则、个案通知、可跟踪申诉和处理结果说明作为内容治理透明度的重要基础。[Santa Clara Principles](https://santaclaraprinciples.org/)
- Mayo Clinic Connect 案例识别出指导、连接、语义整理、行政和执法五类互补治理实践，且部分由社区志愿成员协助完成，说明“社区赋能”不应只等同于处罚权限。[Successful Moderation in Online Patient Communities](https://pmc.ncbi.nlm.nih.gov/articles/PMC7109609/)
- 无障碍验收应以完整流程而非单个控件为对象。WCAG 2.2 特别包含焦点不被遮挡、最小目标尺寸、避免重复输入、可访问认证及状态消息等要求。[WCAG 2.2](https://www.w3.org/TR/WCAG22/)

## 3. 核心用户与待验证需求

### 3.1 访客 / 潜水者

核心任务：快速判断“这里有没有与我相关、可信且近期的讨论”。

需要：无需注册即可读；清晰的节点归属、作者、时间和互动量；稳定的搜索；不被登录弹窗反复打断；公开页不暴露隐私信息。

主要流失点：首屏像空站、节点命名难懂、内容摘要不足、搜索零结果无建议、点击互动后丢失阅读位置。

### 3.2 首次参与者

核心任务：从某个具体帖子或节点完成第一次收藏、反应、关注、评论或发帖。

需要：知道注册价值；登录后恢复原任务；发布前理解规则和不可变后果；第一次贡献后得到明确确认和可预期的下一步。

主要流失点：认证墙没有上下文、注册字段过多、验证邮件无恢复路径、不可变提示过晚、发布失败导致内容丢失。

### 3.3 稳定贡献者

核心任务：持续发现相关讨论、维护草稿、回复他人、沉淀关系并控制通知。

需要：关注 Feed 有明确来源；通知深链接可靠；个人主页能回看贡献；草稿状态可信；互动写入可撤销的仅限收藏/反应/关注等可逆行为。

主要流失点：通知噪声、跟帖上下文丢失、重复提交、草稿冲突、隐藏目标形成死链。

### 3.4 受影响用户 / 举报人

核心任务：快速离开有害互动，阻止继续接触，必要时提供足够信息并跟踪处置。

需要：举报、屏蔽入口可找到但不抢占主任务；知道屏蔽影响范围；知道举报是否成功、处理到哪一步；隐私与防报复预期明确。

主要流失点：原因选项含糊、必须重复描述目标、提交后毫无反馈、屏蔽和举报被混为一谈、治理状态泄露敏感细节。

### 3.5 审核人员

核心任务：基于完整上下文做一致、可复核的最小必要处置。

需要：案件优先级、目标上下文、历史动作、适用规则和用户影响同屏可见；动作互斥关系清楚；必须填写理由；审计记录不可变；申诉交由不同处理者复核（若组织规模允许）。

主要风险：只看单条内容误判、把隐藏当删除、把锁帖和用户限制混用、无理由快捷处罚、并发处理覆盖他人决定。

## 4. 核心任务与旅程

### 4.1 匿名发现到深度阅读

| 阶段 | 用户问题 | 页面/交互 | 成功信号 |
|---|---|---|---|
| 进入 | 这里在讨论什么？ | 首页最新 Feed；真实帖子摘要；节点、作者、时间、计数 | 10 秒内能说出站点主题与内容组织方式 |
| 缩小范围 | 哪些内容和我有关？ | 节点筛选、节点目录、搜索、标签 | 筛选结果有清晰当前条件和重置入口 |
| 判断质量 | 这篇值得读吗？ | 帖子详情、作者公开摘要、节点规则、评论上下文 | 不登录即可读正文与公开评论 |
| 决定参与 | 我想保存或回应 | 收藏/反应/关注/评论入口 | 被拦截时说明登录后会恢复当前任务 |

### 4.2 登录拦截与任务恢复

必须保存的恢复包：`returnUrl`、触发动作、目标 ID、滚动锚点、评论/回复文本、发帖草稿 ID；敏感数据不得放入公开 URL。

```text
受限动作
→ 原位说明“登录后继续”
→ 登录 / 注册 / 验证
→ 回到来源页面和锚点
→ 恢复输入
→ 由用户确认后执行写操作
```

不得在登录成功后自动发布帖子、评论或举报；这些高后果写操作必须再次由用户确认。收藏、反应、关注是否自动续做应由产品统一决定，静态原型建议显示“继续收藏/继续关注”确认，以避免跨页面后的意外状态改变。

### 4.3 草稿到不可变发布

1. 选择有效节点，页面立即展示该节点的相关规则。
2. 输入标题、Markdown、标签和附件；显示草稿已保存/未保存/冲突状态。
3. 预览必须使用与正式发布相同的清洗结果。
4. 发布按钮打开最终确认，展示节点、标题、正文摘要、附件数和“发布后不能编辑或删除”。
5. 提交中禁用重复操作并保留页面内容。
6. 成功进入帖子详情，显示可定位的成功状态；失败留在原页并给出可恢复原因。

### 4.4 评论 / 回复到再次回访

1. 用户从帖子正文或某条评论发起评论/回复。
2. 回复框在目标行内容区展开，并明确“回复给 @用户”和父评论摘要。
3. 提交前就近提示评论不可编辑或删除。
4. 成功后滚动并聚焦新评论；URL 带稳定评论锚点。
5. 被回复者通过通知深链接回到评论上下文。
6. 若目标被隐藏，通知页和详情页显示安全占位，并提供回到帖子或通知列表的路径。

### 4.5 举报、屏蔽和申诉

| 任务 | 必要步骤 | 不应发生 |
|---|---|---|
| 举报 | 核对目标摘要 → 选择原因 → 可选补充说明 → 提交 → 获得案件/状态入口 | 让用户重新复制链接；提交后只给 Toast；暴露举报人身份 |
| 屏蔽 | 展示对象和影响范围 → 确认 → 告知可在设置中解除 | 暗示屏蔽等同于平台处罚；把屏蔽设为不可逆 |
| 查看进度 | 显示已提交、处理中、已处理/关闭等用户可理解状态 | 暴露审核内部备注、被举报人敏感信息或风控信号 |
| 申诉 | 识别原决定 → 展示规则与理由 → 补充材料 → 跟踪 → 收到结果与理由 | 由同一界面直接改写原审计记录；申诉提交后无状态 |

## 5. 常见痛点与设计对策

| 痛点 | 风险 | 对策 | 原型必须体现 |
|---|---|---|---|
| 空 Feed / 空节点像“没人用” | 访客无法判断价值 | 冷启动前用真实账号发布真实、可回复的基础内容；空状态说明如何切换节点/清除筛选 | 最新 Feed、节点页的真实列表与空状态 |
| 节点与标签边界不清 | 选择困难、发错位置 | 节点解释“长期讨论空间”，标签解释“跨节点主题”；发布时节点必选、标签辅助 | 节点选择器帮助文案、搜索分类 Tabs |
| 登录后丢上下文 | 首次参与转化断裂 | 保存来源、锚点、输入与意图；恢复后明确下一步 | 登录页来源提示、恢复后的原位草稿 |
| 不可变规则造成焦虑 | 不敢发或误发后强烈挫败 | 提前提示、自动草稿、预览、最终确认、双重防重复 | 发帖/评论确认态和失败态 |
| 评论层级不清 | 回复错人、无法理解对话 | 显式“回复给”、父评论摘要、内容缩进但分隔线全宽、稳定锚点 | 多层回复样例和隐藏评论占位 |
| 互动没有反馈 | 用户不确定是否成功 | 原位计数/状态更新、适度 live region、失败可重试 | 收藏/反应/关注的成功、失败、429 |
| 通知过载 | 关闭全部通知或离开 | 默认按价值分型；偏好按事件类型控制；已读操作幂等 | 通知筛选、通知设置、全部已读 |
| 举报像黑箱 | 降低信任，重复举报 | 显示规则、允许补充说明、提交凭证、进度和结果类别 | 举报成功页/状态页，不只弹窗 |
| 治理动作破坏对话上下文 | 引用和通知变死链 | 隐藏而非删除；保留安全 tombstone、评论位置和返回路径 | 帖子/评论隐藏、锁帖、404 降级 |
| 错误被统称“网络错误” | 用户不知道能否恢复 | 区分字段错误、401、403、404、409、429、局部失败 | 页面级和模块级错误样例 |

## 6. 信任与安全体验原则

### 6.1 规则可见且与任务相关

- 节点页：展示完整节点规则和发帖状态。
- 发布页：展示当前节点规则摘要及完整规则入口。
- 举报页：原因旁提供规则解释，不要求用户先理解审核术语。
- 处置通知：说明被采取的动作、对应规则、持续时间、影响范围和申诉入口。
- 审核案件：展示“原始内容只读快照 + 对话上下文 + 适用规则 + 历史动作”，动作区不得提供编辑或删除原文。

### 6.2 举报与屏蔽职责分离

- 举报解决“请平台判断并处置”；屏蔽解决“我现在不想再看到/接触此用户”。
- 举报提交后可建议屏蔽，但不得默认替用户执行。
- 屏蔽确认必须说明对 Feed、搜索、提及、通知的影响，以及不能阻止哪些公开可见场景；最终文案以真实契约为准。
- 举报记录只显示举报人需要知道的状态，不展示内部证据、队列人员或被举报人的私密处置细节。

### 6.3 最小必要处置与可申诉

治理动作在 UI 上明确分为：内容隐藏/恢复、帖子锁定/解锁、用户警告/限流/禁言/封禁/恢复、申诉决定。每个动作必须显示影响范围和持续时间，要求理由，并产生不可变审计记录。

高风险确认对话框需包含目标、动作、持续时间、理由摘要和不可逆/可恢复属性；按钮使用具体动词，例如“锁定评论”“隐藏此帖子”，不用笼统“确认”。

### 6.4 安全降级

- 隐藏帖子/评论：保留位置和“因社区规则暂不可见”的最少说明，不泄露原文。
- 无权限：普通权限不足显示 403；涉及隐私或对象存在性时按契约返回 404。
- 通知目标失效：通知仍可读，但深链接落到安全状态并提供返回路径。
- 账户安全通知：与普通互动通知在视觉和文案上区分，提供直接检查会话/改密路径。
- 举报、申诉、发布、评论均需幂等，重复点击不生成重复记录。

## 7. 冷启动与社区赋能机制

以下机制均在当前 P0/P1 范围内实施，不新增推荐、运营精选、徽章、活动、私信或节点管理功能。

### 7.1 冷启动内容策略

1. **先少而活，不多而空。** 初始节点控制在能被持续照看的数量；节点仍通过 SQL 初始化，不增加管理 UI。
2. **每个节点准备三类真实内容：** 新手可回答的问题、可复用的知识帖、能引发观点交流的开放讨论。以普通帖子发布，不增加“精选”标签或特殊推荐位。
3. **不使用假用户、假计数或占位讨论。** 种子内容由可识别的真实运营/社区账号发布，并允许真实回复。
4. **搜索零结果转向可行动路径：** 建议清除筛选、尝试同义词、浏览相关节点；登录用户可从当前查询带入发布草稿，但发布前仍需选择有效节点。
5. **冷启动期人工保障首响。** 社区维护者通过现有 Feed、节点关注和通知发现新内容并给出有信息量的回复；不需要新增前台角色或功能。

### 7.2 从阅读者到贡献者的参与阶梯

```text
公开阅读/搜索
→ 收藏或反应
→ 关注用户/节点
→ 评论/回复
→ 发布主题
→ 举报不当内容
→ 在现有权限体系下参与治理
```

界面不应强迫每个人走完整阶梯。收藏、关注和屏蔽属于用户控制，不应包装成成长积分任务；现有经验/金币仅可作为真实能力展示，不扩展成徽章、等级或奖励系统。

### 7.3 社区赋能的可执行做法

- **让贡献可被发现：** 公开用户页展示其公开帖子和评论；关注 Feed 解释每条内容因关注了哪个用户或节点而出现。
- **把回应变成习惯：** 冷启动运营目标优先监控“首帖是否获得有效回复”，而非总发帖量。
- **用规则教育代替只处罚：** 锁帖/隐藏/警告通知给出具体规则和可行的下一次行为；申诉结果说明理由。
- **支持社区协助而非泛化执法：** 鼓励成员通过回复补充上下文、引导去正确节点和举报风险；治理权限仍由后端权限控制。
- **照顾治理人员：** 审核队列显示优先级与并发状态；案件详情保留上下文；危险动作确认；审计和申诉复核降低个人决策压力。

### 7.4 建议指标

| 目标 | 核心指标 | 诊断指标 |
|---|---|---|
| 证明公开阅读价值 | 匿名帖子详情到达率、7 日匿名回访 | 搜索零结果率、节点筛选使用率、详情退出位置 |
| 提升首次参与 | 受限动作登录恢复完成率 | 登录拦截流失、验证邮件恢复率、恢复输入成功率 |
| 改善新人体验 | 首次贡献 24 小时有效回复率、首次贡献者 7/30 日回访 | 首次回复时间、首次贡献失败率、发布确认取消率 |
| 形成关系回访 | 关注 Feed 7 日使用率、通知深链接成功率 | 通知分类点击/关闭率、隐藏目标降级率 |
| 建立治理信任 | 举报完成率、状态查看率、申诉处理时长 | 举报重复率、原因分布、处置恢复率、429 率 |

指标不得记录邮箱、私密正文或未清洗输入；公开页面埋点不应依赖私有 `viewerState`。

## 8. 全页面与关键交互建议

### 8.1 全局导航与路由

- 匿名导航：首页、节点、搜索、登录、注册。
- 登录导航：首页、节点、关注、搜索、发布、通知、用户菜单。
- 治理入口只对有权限用户出现；前端隐藏入口不能替代后端鉴权。
- 移动端 Drawer 打开后焦点进入菜单，Escape 关闭并恢复到触发按钮；当前路由有文本与非颜色状态。
- 每页只有一个明确主操作；普通导航和当前项不滥用主色。

### 8.2 页面级建议

| 页面 | 首要任务 | 关键建议 / 必备状态 |
|---|---|---|
| 首页 / 最新 Feed | 判断社区价值并进入内容 | 节点筛选、连续列表、匿名可读；loading/empty/partial/error/分页 |
| 节点目录 | 理解内容分区 | 每行名称、描述、状态；不出现新增/编辑/排序/删除 |
| 节点详情 | 阅读规则、关注、浏览帖子 | 规则、发帖能力、关注状态、帖子列表；锁定/不可发状态 |
| 帖子详情 | 阅读、互动、参与讨论 | 正文与元信息、反应/收藏/关注、评论框、评论列表；锁帖/隐藏/404/429 |
| 发布 / 草稿 | 安全完成不可变发布 | 保存状态、预览、附件、节点规则、最终确认；dirty/conflict/submitting/failed |
| 搜索 | 找到特定对象 | 帖子/用户/节点/标签 Tabs、筛选摘要、清除条件、类型化零结果 |
| 标签页 | 浏览跨节点主题 | 标签解释、帖子列表、筛选和分页；不伪装推荐页 |
| 公开用户页 | 判断身份和公开贡献 | 资料、帖子、评论、关注/粉丝、关注/屏蔽；绝不出现邮箱 |
| 登录 / 注册 | 建立会话并继续原任务 | 来源任务提示、字段级错误、账户枚举安全、返回原页 |
| 邮箱验证 | 完成注册闭环 | 发送状态、重发倒计时/429、更换邮箱或返回登录的真实路径 |
| 忘记 / 重置密码 | 恢复账户 | 不泄露账户是否存在；令牌失效、成功、重新请求路径 |
| 关注 Feed | 查看关系内容 | 每项解释来源；空状态引导去节点/用户页关注 |
| 收藏 | 回看保存内容 | 连续列表、取消收藏、目标隐藏的安全占位 |
| 通知 | 处理需要回访的事件 | 分类筛选、单条/全部已读、深链接、失效目标降级 |
| 本人主页 | 回看身份与贡献 | 私有概要但不展示邮箱；主页/设置/草稿等真实入口 |
| 设置：资料 | 更新公开身份 | 预览公开效果、字段验证、保存冲突 |
| 设置：隐私 | 控制可见与互动 | 每项解释影响范围和默认值 |
| 设置：通知 | 控制事件通知 | 按回复、提及、关注、互动、审核、安全分型 |
| 设置：安全/会话 | 管理密码与设备 | 当前会话标识、撤销确认、撤销后的结果和重新登录 |
| 草稿列表 | 找回未发布内容 | 更新时间、节点、保存状态；可继续或放弃草稿 |
| 屏蔽列表 | 查看和解除个人屏蔽 | 影响说明、解除确认与结果 |
| 举报提交 | 提供平台判断信息 | 目标摘要、原因、说明、规则、成功状态；重复提交幂等 |
| 举报进度 | 确认平台已处理 | 用户可懂状态、时间线、结果类别；不泄露内部审核信息 |
| 审核队列 | 确定处理顺序 | 筛选、优先级、状态、分页、并发提示；唯一表格/列表外框 |
| 案件详情 | 做最小必要处置 | 上下文、规则、历史、动作确认、理由、冲突；无编辑/删除原文 |
| 申诉提交/进度 | 请求复核并获知结果 | 原决定、理由、补充材料、状态、结果说明 |
| 审计日志 | 追溯治理动作 | 只读、筛选、分页、动作详情；任何用户都不能改写 |
| 401 / 403 / 404 / 429 | 解释并提供下一步 | 各自专用文案和可恢复动作，不统一成网络错误 |

## 9. 可用性与无障碍验收标准

### 9.1 关键流程验收

- [ ] 匿名用户从首页进入任一公开帖子并阅读评论，全程不要求登录。
- [ ] 匿名用户在评论框输入不少于 100 字后登录，登录成功返回同一帖子和评论锚点，输入完整保留。
- [ ] 注册流程包含邮箱验证时，完成验证后仍能恢复最初任务；刷新或后退不会重复发布写操作。
- [ ] 发帖最终确认同时展示节点、标题、正文摘要、附件数和“发布后不能编辑或删除”；取消后内容不丢失。
- [ ] 双击发布/评论/举报只生成一条记录；提交中有明确状态。
- [ ] 发布成功后的帖子详情，以及评论成功后的评论行，均没有编辑、删除、撤回入口。
- [ ] 评论失败、附件失败、草稿冲突和 429 均保留用户输入并给出原位恢复动作。
- [ ] 通知深链接可定位具体评论；目标隐藏、无权限或不存在时不会进入空白页或循环跳转。
- [ ] 举报提交后显示可追踪状态；重复提交返回同一或明确的已有举报结果。
- [ ] 屏蔽确认解释影响，解除屏蔽可在设置中完成；屏蔽不冒充平台处罚。
- [ ] 治理动作只包含隐藏/恢复、锁定/解锁和用户处置；不存在编辑、删除、替换原内容。
- [ ] 申诉能关联原决定、提交补充信息并查看结果；审计日志保持只读。

### 9.2 页面完整性验收

- [ ] 第 8.2 节全部页面均有可跳转入口或从相关任务可达，没有孤立页面。
- [ ] 所有按钮在静态原型中至少完成页面跳转、状态切换、展开/关闭或表单验证反馈；无无响应按钮。
- [ ] 不出现私信、推荐/热门算法、运营精选、活动、徽章扩展、会员、钱包、订单、支付或节点管理入口。
- [ ] 所有公开页面只使用公开用户信息；邮箱仅出现在本人必要的安全/账户设置。
- [ ] 页面标题、可见品牌、canonical 示例统一为 X2Post / `https://x2post.com`。
- [ ] 私有页和治理页原型标明 `noindex`，公开 Feed、节点、帖子和用户页具备可索引结构。

### 9.3 状态完整性验收

- [ ] Feed、节点、帖子、评论、搜索、通知、草稿、举报、审核等一级模块均能演示 loading、success、empty、partial 和 error。
- [ ] 全局或页面样例分别演示 401、403、404、429，不以同一“网络错误”替代。
- [ ] 写表单演示 dirty、validating、submitting、conflict、failed；刷新/返回前对未保存输入给出提示。
- [ ] 右栏加载失败不阻塞左侧核心任务；右栏无真实内容时自然省略。

### 9.4 无障碍与响应式验收

- [ ] 每页唯一 `h1`，标题层级连续；使用 `header/nav/main/aside/section/article/form/list` 等恰当语义。
- [ ] 键盘可完成注册、登录、发布、评论、举报、屏蔽、审核和申诉完整流程。
- [ ] Drawer、Dropdown、Dialog 的焦点进入、限制、Escape 和返回均正确；焦点不被固定导航遮挡。
- [ ] Tabs 支持方向键并通过 ARIA 关联面板；动态结果和保存/提交状态可被读屏感知但不强抢焦点。
- [ ] 触控目标至少 44×44px；普通文字对比度至少 4.5:1；状态不只依赖颜色。
- [ ] 320px 宽度、200% 缩放下核心任务可完成且无整页横向滚动；表格/代码仅自身可横向滚动。
- [ ] 评论层级同时使用文字和缩进表达，不能只依靠视觉位置。

## 10. 上线后验证计划

在正式应用阶段，建议以两轮研究验证桌面研究假设：

1. **上线前任务测试（每类 5–8 人）：** 匿名读者、首次贡献者、稳定贡献者、审核人员分别完成核心任务。重点观察是否理解节点/标签、是否预期登录恢复、是否注意不可变规则、是否分得清举报与屏蔽。
2. **上线后 2–4 周混合研究：** 用第 7.4 节指标定位漏斗，再访谈“完成首次贡献后未回访”“举报后反复查看”“关闭通知”三类用户，避免只访谈最活跃成员。

建议的高优先级可用性问题：

- 用户能否在 10 秒内解释首页内容组织方式？
- 匿名用户是否相信登录后内容不会丢？
- 用户是否在点击最终发布前准确复述“不可编辑或删除”？
- 首次发帖者是否知道下一步从哪里查看回复？
- 用户能否准确说明“屏蔽”和“举报”的差异？
- 被处置用户能否找到原因、时限和申诉路径？
- 审核人员能否在不编辑/删除原文的前提下完成全部处置？
