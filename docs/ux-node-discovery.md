# X2Post Feed 节点发现专项 UX 建议

> 日期：2026-07-13  
> 范围：首页 / Feed 顶部的节点发现与筛选  
> 依据：当前静态原型、`docs/ux-research.md`、产品方案与 UI 设计规范  
> 约束：节点由 NodeApi 只读提供；不新增节点管理、推荐/热门、运营精选或 P2/P3 能力

## 1. 结论

推荐采用 **“可见节点快捷筛选 + 全部节点入口 + 长列表抽屉/搜索”** 的渐进式方案：

- **当前只有 5 个节点时，桌面和移动端都应直接显示全部节点名称。** 不再把移动端唯一的筛选入口放进 `select`。
- **桌面：** 在 Feed 列表上方设置独立的“按节点浏览”筛选条，依次显示“全部、产品设计、前端开发、社区运营、写作交流、生活方式”，末尾提供“了解全部节点”。
- **移动：** 同一组节点使用两行或自然换行的可点击项，保持全部选项可见；不以横向滚动作为当前 5 个节点的默认方案。
- **节点数量增长后：** 首屏保留“全部”和按 NodeApi `sort` 排序的 3–5 个稳定快捷项，再提供带数量的“全部节点（N）”按钮；按钮打开节点 Drawer，长列表内再提供搜索。
- **节点目录仍负责理解节点，Feed 控件只负责缩小内容范围。** 帖子行中的节点名继续链接到节点详情页，避免把“过滤帖子”和“阅读节点规则”混成一个动作。

推荐组件语义是项目级 `NodeFilterBar`：内部可用 daisyUI `Menu` / `Button` 视觉，但应以普通链接导航表达，不使用 Tabs、ARIA `menu` 角色或 daisyUI `Filter` 的“选中后隐藏其他选项”行为。长列表回退使用项目 `Drawer` + `Input` + `List`。

## 2. 当前原型审计

### 2.1 已发现的问题

原始 Feed 把紧凑 `select` 放在 Panel 标题右侧，带来四个问题：

1. **选项不可见：** 用户打开前只能看到当前值，无法快速形成“社区有哪些节点”的认知。
2. **视觉权重过低：** 小尺寸表单控件与 Panel 标题并列，更像排序或次要设置，不像主要发现入口。
3. **选择成本偏高：** 打开控件后才能扫描，再选择；在移动端还受系统选择器表现影响。
4. **节点理解不足：** 只有名称，没有描述、活跃度或进入规则页的显式路径。

### 2.2 当前工作区的中间改版

检查时，原型已经出现一版未提交的改进：

- `prototype/assets/app.js:34–45` 增加节点快捷导航和右栏节点发现模块。
- `prototype/assets/app.js:86–95` 已根据 `#/feed?node={slug}` 真实过滤帖子，并补充空状态和清除筛选。
- 桌面端显示横向 `Menu`，移动端仍回退到 `select`。
- Feed 右栏显示前三个节点，但移动单栏时右栏位于核心内容之后，不能替代顶部发现入口。

这版已经解决“筛选没有行为”和桌面选项完全隐藏的问题，但仍需优化：

- `sm` 以上强制单行且依赖横向滚动，窄平板/200% 缩放时会隐藏末端节点，当前没有明显滚动提示。
- `<640px` 仍只有 `select`，移动端发现性没有实质提升。
- 右栏前三个节点与顶部筛选重复，并且截断到 3 个可能造成节点重要性误读；右栏更适合放简短“节点是什么”说明和“节点目录”入口。
- 动态筛选后虽然有可见摘要，但还需要可靠的状态播报、焦点和浏览器历史规则。

## 3. 方案比较

评分：5 为最好；“选择成本”分数越高代表越省步骤。

| 方案 | 可发现性 | 选择成本 | 响应式 | 无障碍稳健性 | 适用情况 | 结论 |
|---|---:|---:|---:|---:|---|---|
| 原生 `select` | 2 | 3 | 5 | 4 | 很长且低频的单选列表 | 不作为 Feed 主要入口 |
| 全部可见、自然换行的快捷项 | 5 | 5 | 4 | 5 | 3–7 个短名称节点 | **当前首选** |
| 横向滚动快捷项 | 3 | 5 | 4 | 3 | 选项略多且高度极受限 | 次选；需明确溢出提示 |
| Dropdown / Menu | 2 | 3 | 5 | 3 | 次要命令或不常用切换 | 不适合承担冷启动发现 |
| Drawer + 节点列表 | 3 | 3 | 5 | 4 | 8 个以上，需描述和搜索 | 长列表回退方案 |
| 节点搜索 | 2（浏览）/5（已知目标） | 2 | 5 | 4 | 用户知道名称、节点较多 | Drawer/目录辅助，不单独使用 |
| 完整节点目录 | 5 | 2 | 5 | 5 | 学习差异、规则与活跃度 | 保留为二级探索页 |

GOV.UK Design System 建议在面向公众的服务中把 `select` 作为最后手段，因为一些用户觉得它很难使用；在使用前应先尝试减少选项或改用让选项可见的方案。[GOV.UK Select guidance](https://design-system.service.gov.uk/components/select/)

横向滚动不是完全禁止，但它会使视口外选项缺少天然可见性。类似的移动端研究也观察到，横向 Tabs 会让部分入口默认隐藏，从而增加遗漏风险；这里将该结论作为对节点筛选的类比，而不是直接等同。[Baymard: Avoid Horizontal Tabs](https://baymard.com/blog/avoid-horizontal-tabs)

## 4. 推荐交互模型

### 4.1 语义：过滤 Feed，不是进入节点页

顶部快捷项改变 Feed 视图：

```text
全部                → #/feed
产品设计            → #/feed?node=product
前端开发            → #/feed?node=frontend
了解全部节点        → #/nodes
帖子行里的“产品设计” → #/nodes/product
```

这样能明确区分：

- 快捷项回答“我只想看哪个节点的帖子？”
- 节点详情回答“这个节点是什么、有哪些规则、是否要关注？”

筛选状态必须进入 URL，支持刷新、分享、浏览器前进/后退和从帖子详情返回时恢复。节点顺序来自 NodeApi 的 `sort`，前端不按热度或个性化重新排序，避免引入冻结的推荐逻辑。

### 4.2 桌面布局（≥1024px）

推荐结构：

```text
最新帖子                                      发布帖子
先从节点缩小范围，再进入一段具体讨论。

┌──────────────────────────────────────────────────────┐
│ 按节点浏览                         了解全部节点       │
├──────────────────────────────────────────────────────┤
│ [全部] [产品设计] [前端开发] [社区运营] [写作交流]   │
│ [生活方式]                                           │
└──────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────┐
│ 产品设计 · 最新帖子                     清除筛选     │
├──────────────────────────────────────────────────────┤
│ 帖子列表……                                           │
└──────────────────────────────────────────────────────┘
```

- 筛选条是独立一级 Panel，位于页面说明和 Feed 列表之间，不缩回 Feed 标题右上角。
- 6 个选项能放一行时放一行；空间不足时自然换行，不默认横向滚动。
- 选中项使用边框/底色/字重和文本状态共同表达，且只保留一个当前项。
- “了解全部节点”是普通链接，不与筛选项混在同一单选组中。
- 右栏不再重复前三个筛选项；可显示一句节点解释、关注节点数或节点目录入口。

### 4.3 平板布局（768–1023px）

- 保持顶部独立 Panel 和可见快捷项；允许两行。
- 不把双栏压缩进平板，符合主 UI 文档的单栏要求。
- 若名称本地化后过长，可将“全部节点（N）”放在第二行末尾，但不能只保留一个下拉。

### 4.4 移动布局（<768px）

当前 5 个节点的推荐方案：

```text
按节点浏览          全部节点
[全部] [产品设计] [前端开发]
[社区运营] [写作交流] [生活方式]
```

- 快捷项自然换行，内容决定宽度；320px 下允许 3 行，不牺牲可发现性。
- 点击后保持焦点在选中项附近，列表原位更新；不要自动把页面滚到顶部。
- 页面下方出现筛选结果摘要，如“当前显示：产品设计 · 12 篇帖子”。
- 全部节点按钮进入 `/nodes`；当节点数增长时再改为打开 Drawer。

不建议当前使用横向滚动。若未来因节点增多必须使用：

- 最右侧必须露出下一项的一部分或提供明确的“横向滚动查看更多”视觉提示。
- 保留一个始终可见的“全部节点（N）”按钮，不让完整发现依赖滑动。
- 页面载入与浏览器返回时将当前项滚入视口，但不强制抢焦点。
- 不隐藏滚动区域中的键盘焦点；所有选项仍按合理 DOM 顺序可达。

### 4.5 节点较多时的 Drawer 与搜索

启用建议：节点超过 7 个，或本地化名称导致 3 行以上时。

Drawer 内容：

1. 标题“选择节点”，显示节点总数。
2. “全部帖子”作为第一项，并显示当前选择。
3. 节点列表显示名称 + 一行描述；可选显示公开帖子数，不用关注数做默认排序。
4. 节点达到 8 个时显示搜索；搜索匹配名称和描述。
5. 零结果说明“没有匹配节点”，提供清除搜索。
6. 选择后关闭 Drawer、更新 URL 和 Feed，并把焦点恢复到“全部节点”触发按钮。

Drawer 不能塞进全局导航 Drawer，也不应使用 hover-only Dropdown。两者任务不同：全局 Drawer 用于站点导航，节点 Drawer 用于当前 Feed 的单一筛选任务。

## 5. 组件与无障碍建议

### 5.1 推荐组件映射

| 场景 | 项目组件 | daisyUI 内部候选 | 说明 |
|---|---|---|---|
| 快捷筛选 | `NodeFilterBar` | `Menu` + `Button` 视觉 | 普通链接列表，支持 URL 与无 JS 导航 |
| 当前状态 | `NodeFilterLink` | active modifier | 使用 `aria-current="page"`，不只靠颜色 |
| 全部节点 | `ButtonLink` | Button | 独立于筛选组 |
| 长列表选择 | `NodePickerDrawer` | Drawer + Menu/List | 带焦点进入、限制、Escape 和恢复 |
| 节点搜索 | `Input` | Input | 可见 label；只在长列表出现 |
| 更新反馈 | `FeedResultStatus` | Alert/状态文字 | `aria-live="polite"`，不抢焦点 |

不推荐直接使用 daisyUI `Filter`：其标准行为会在选中后隐藏其他选项并显示重置按钮，会再次降低其他节点的可发现性。也不推荐 Tabs，因为 WAI-ARIA Tabs 是一组控制相应 tabpanel 的复合组件，需要方向键和 `aria-controls` 等完整行为；Feed 的 URL 过滤更接近导航视图链接。[WAI-ARIA Tabs pattern](https://www.w3.org/WAI/ARIA/apg/patterns/tabs/)

### 5.2 键盘与读屏

- 推荐使用真实 `<a href>`；筛选容器为有可见标题的 `<nav>`，当前链接设置 `aria-current="page"`。
- 不给普通网站导航添加 ARIA `menu/menuitem` 角色，避免引入桌面应用式键盘预期。
- 若实现为原生 radio，则必须放入有 `legend` 的 `fieldset`，并遵循箭头键切换的 radio group 模式。[WAI-ARIA Radio Group pattern](https://www.w3.org/WAI/ARIA/apg/patterns/radio/)
- 筛选完成时礼貌播报“已显示产品设计节点，共 12 篇帖子”；不把焦点强制移动到列表。
- 加载失败时保留筛选条和当前项，在 Feed Panel 内原位显示重试。
- 所有触控目标按项目 UI 规范至少 44×44px；WCAG 2.2 AA 的最低基线为可容纳 24×24 CSS px 或满足间距例外，但 X2Post 应坚持更宽松的 44px 项目标准。[WCAG 2.2 Target Size](https://www.w3.org/WAI/WCAG22/Understanding/target-size-minimum.html)

## 6. 状态与边界

| 状态 | 预期行为 |
|---|---|
| 默认 | “全部”当前，显示全 Feed，节点名称全部可见 |
| 已筛选 | 当前节点明显标识；标题、结果数和 URL 同步 |
| Loading | 保留筛选条；Feed Panel 内 Skeleton；当前项不闪回“全部” |
| Empty | “此节点暂时没有公开帖子”；提供查看节点和清除筛选 |
| Partial | 保留已加载帖子，说明部分失败并局部重试 |
| Error | 不清空当前筛选；原位错误与重试；保留 URL |
| 无效 slug | 安全回到“全部”并说明筛选已失效，或显示可恢复 404；需与路由契约统一 |
| 节点关闭发帖 | 仍可按契约阅读历史内容；节点详情说明状态；Feed 筛选不显示发布能力假设 |
| 返回 Feed | 恢复 node、页码和滚动位置 |
| 200% 缩放 | 快捷项换行，不出现整页横向滚动 |

## 7. 验收标准

### 7.1 可发现性与选择成本

- [ ] 首次进入 Feed 时，无需点击即可看见“按节点浏览”以及当前全部 5 个节点名称。
- [ ] 用户从 Feed 首屏最多一次点击即可切换到任一当前节点。
- [ ] 用户最多一次点击进入完整节点目录。
- [ ] 当前节点不只靠颜色表达，并同时反映在 URL、列表标题和结果摘要中。
- [ ] 节点顺序来自 NodeApi `sort`，刷新后稳定；没有按热度或个性化排序。

### 7.2 行为

- [ ] 每个快捷项真实过滤帖子；“全部”和“清除筛选”恢复完整 Feed。
- [ ] 选择节点后页码重置到第一页；浏览器前进/后退准确恢复节点与页码。
- [ ] 从帖子详情返回时恢复 Feed 的节点筛选和滚动位置。
- [ ] 复制 `#/feed?node={slug}` 在新窗口打开可得到同一筛选视图。
- [ ] 重复选择当前节点不发起不必要请求，不产生重复历史记录。
- [ ] Empty、Error、429 均保留筛选控件和当前状态，用户可清除或重试。

### 7.3 响应式

- [ ] 1024px 以上优先单行；放不下时换行，不截断节点名。
- [ ] 768–1023px 为单栏，筛选条位于 Feed 列表之前。
- [ ] 320px 下所有当前节点均直接可见，可换成 2–3 行；整页无横向滚动。
- [ ] 200% 缩放下节点项仍可点击，焦点不被固定 Navbar 遮挡。
- [ ] 若未来启用横向滚动，存在明确溢出提示、完整节点按钮，并自动让当前项进入可视区。

### 7.4 无障碍

- [ ] 筛选区域有可见名称“按节点浏览”，读屏可识别当前项。
- [ ] 真实链接可使用 Tab/Shift+Tab、Enter 操作；焦点样式清楚且不被裁剪。
- [ ] 筛选结果通过 `aria-live="polite"` 播报，不强制移动焦点。
- [ ] 所有触控目标至少 44×44px，目标间距足够，状态对比符合 WCAG 2.2 AA。
- [ ] Drawer 打开后焦点进入，Escape/遮罩可关闭，关闭后焦点回触发按钮；背景不可误操作。
- [ ] 搜索输入有可见 label，零结果与清除搜索均可被键盘和读屏访问。

### 7.5 建议验证指标

上线后比较改版前后 2–4 周：

- 节点筛选触达率：进入 Feed 后使用任一节点筛选的会话占比。
- 节点覆盖率：被选择过的节点数 / 可见节点数，监控是否只触达首项。
- 节点详情到达率：从 Feed 筛选条或帖子元信息进入节点页的比例。
- 筛选后帖子详情点击率与返回保留率。
- 移动端“全部节点”打开率、搜索使用率、搜索零结果率。
- 无效/重复筛选请求率和筛选接口失败率。

首轮可用性测试建议至少覆盖 5 名桌面用户和 5 名移动用户，任务是“找到前端开发节点最近的帖子”“了解生活方式节点的规则”“返回全部帖子”。通过标准：90% 无提示完成；当前 5 节点任一目标的中位选择时间不超过 5 秒；没有参与者把筛选快捷项误认为发帖节点管理。
